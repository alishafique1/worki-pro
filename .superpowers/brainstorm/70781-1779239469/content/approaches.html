3 Ways to Execute the Bark Flip

All three put the request first — they differ in when auth happens and how complex the build is

A

Pure Guest Wizard — auth at the very end

Homepage → pick category → all service questions → location → then enter email + OTP → account created + request saved in one shot.

Pros

  • Maximum conversion — zero friction before value
  • Exactly how Bark works
  • User sees the full wizard before committing

Cons

  • Biggest build — guest state must survive to account creation
  • Need to wire OTP into the end of the request wizard
⭐ Recommended — highest conversion, true Bark model
B

Email-first, then questions

Homepage → pick category → enter email (one field only) → service questions → OTP verify at end → account + request created.

Pros

  • Lead captured early — even if they abandon mid-wizard
  • Simpler state management (email anchors the session)
  • Easy to re-engage via email if they drop off

Cons

  • One extra field before wizard starts
  • Slight friction vs pure guest flow
Good middle ground — GHL gets a lead even on abandonment
C

Category pages → existing OTP → smart onboarding

Homepage → category cards → service detail page with rich CTA → existing OTP login → onboarding wizard is now service-specific (not generic).

Pros

  • Least code change — reuses existing auth flow
  • SEO-friendly — rich service detail pages
  • Faster to ship

Cons

  • Still auth-before-request — doesn't fully solve the problem
  • Homepage still needs a rebuild
Safer scope — but leaves the core friction in place
What all three share
  • Homepage rebuilt with search + visual category grid (no more confusing nav)
  • Service detail pages get real content (pricing, FAQs, local trust signals)
  • Onboarding wizard becomes service-specific with dynamic follow-up questions
  • Provider onboarding stays roughly the same (separate flow)
